# アイアルク ブログ/クラファン制作補助アプリ 実装仕様書

> この文書は、VS Code上のClaude Codeが本リポジトリで実装作業を行う際の仕様書です。
> GitHub Actionsでのデプロイを前提とした構成・ファイル配置を想定しています。

## 0. スコープ（重要）

会社アカウントでの作業のため、**今回の実装範囲はブログ記事生成とクラウドファンディング本文生成の制作補助に限定**します。以下は将来構想として設計には織り込みますが、**今回は実装しません**。

- YouTube解説動画の台本・スライド生成
- 漫画形式の出力
- 小説形式の出力
- 動画生成

実装対象は次の一連の流れのみです。

口述筆記の入力・記録 → 要点抽出 → ブログ記事 or クラファン本文への整形 → DALL-Eによるイメージ画像・スライド風画像の生成

**最小構成（MVP）に必ず含めるもの:**

- 音声入力（Web Speech API）
- 要点抽出・ブログ/クラファン本文生成（Claude API 実呼び出し）
- **画像生成（OpenAI画像API 実呼び出し）** — ブログの挿絵はコンセプトの一部であり、プレースホルダー画像を返すダミー実装は最小構成として不可。プロンプトたたき台生成（Claude）→ 本人が編集 → 画像生成、まで配線すること
  - 注: DALL-E 3 は廃止済み。後継の gpt-image 系を使用する（環境変数 `IMAGE_MODEL`、既定 `gpt-image-1`。画質は `IMAGE_QUALITY`、既定 `medium`）。本文書の「DALL-E」表記は「OpenAI画像API」と読み替える
  - gpt-image系はbase64で画像を直接返すため、一時URL失効の問題（5.5節）はローカル保存で自然に解消される

## 1. システム構成

```
[フロントエンド: HTML + TypeScript（べた書き、フレームワーク不使用）]
   ├─ 入力画面（タイムライン風UI、詳細は5章）
   ├─ 下書き一覧画面
   ├─ 生成結果編集画面（ブログ/クラファン切替）
   └─ 画像生成画面（DALL-Eプロンプト編集UI）
        │ HTTPS (fetch)
        ▼
[バックエンド: Cloud Run / Python / FastAPI]
   ├─ /drafts           下書きCRUD
   ├─ /generate/blog     ブログ整形 (Claude API)
   ├─ /generate/crowdfunding  クラファン本文整形 (Claude API)
   └─ /generate/image    画像生成 (DALL-E API)
        │
        ▼
[Firestore]
   ├─ tenant_settings
   ├─ drafts
   ├─ generated_texts
   └─ generated_images
```

フロントエンドはFirebase Hostingで配信、バックエンドはCloud Runにデプロイする想定。GitHub Actionsで両方のデプロイを自動化する。

2段階構成：(1) 要点抽出 → (2) 出力フォーマットへの整形。`{変数}` は3.0節の `tenant_settings` から埋め込む。

### 4.1 要点抽出プロンプト（`backend/prompts/extract_points.py`）

```
以下は口述筆記された原文です。話し言葉特有の言い淀みや脱線を整理し、
箇条書きで要点を抽出してください。

# 出力形式
- 主題（1行）
- 要点（箇条書き、5〜8個）
- 印象的なエピソードや具体的な発言があれば、そのまま引用候補としてリストアップ

{raw_text}
```

## 5. フロントエンドUI仕様（ワークフロー全面改訂 2026-07-03）

### 5.0 全体フロー

「発想を刺激しながら口述し、筆者が構成を決め、AIが清書し、選んだセクションをスライド化する」流れを3画面+書庫で構成する。

```
口述スタジオ（話す） → セクション構成（筆者が組み立てる） → 記事スタジオ（仕上げ・スライド化） → エクスポート
                                    ↑ いつでも「書庫」から途中再開できる
```

実装方針: フロントは引き続きフレームワーク不使用のTS。画面はハッシュルーティング等で分割する。並び替えはMVPではドラッグ&ドロップではなく↑↓ボタンで実装してよい。

### 5.1 口述スタジオ（入力画面）

- 音声認識（Web Speech API）またはテキスト直接入力で、発話を**上から下に流れるチャット風のタイムライン**として表示する。
- 各発話の確定ごとに、断片を画面下部から薄く浮かび上がらせ、少し経ってから不透明度を下げて「過去のもの」として残す（チップ表示）。
- **沈黙検知トリガー**: 発話が止まってから2〜3秒経過したら、それまでの未処理の断片をまとめてバックエンドに送信する。毎発話ごとにAPIを呼ばない。
- **要点・引用候補パネル**: 抽出された要点を画面右側にリアルタイムで追記表示する。印象的な発言は強調色で「引用候補」として表示する。
- **問いかけパネル（発想刺激の核）**:
  - 発話内容をもとに、AIが深掘りの問いかけ（例:「そのとき子どもたちは何と言っていましたか？」）を生成し、タイムラインの脇に自然に浮かび上がらせる。
  - 筆者はそれを見て答えて話してもいいし、**無視して進んでもいい**。回答を要求するUI（ボタン等）は付けない。
  - タイムラインと連動し、新しい問いかけが下から現れ、古い問いかけは上へスクロールアウトしていく。
  - 問いかけ生成は沈黙検知トリガーの要点抽出と同一のAPI呼び出しで行い、呼び出し回数を増やさない。
- 材料が溜まったら「構成をつくる」ボタンで次の画面へ。

### 5.2 セクション構成画面（筆者が組み立てる）

- 要点をもとにAIが**セクション構成案**（各セクション = タイトル + 要旨1〜2行、6〜10個程度）を生成して提示する。
- 各セクションに**チェックボックス**があり、筆者が採用/不採用を決める。
- **↑↓ボタンで順番を入れ替え**られる。構成の主導権は筆者にある。
- 「この構成で記事にする」ボタンで、**採用セクションと順序を確定してから**本文生成に進む。AIは確定した構成に従って書く（構成を勝手に変えない）。
- 確定した構成（採否・順序含む）は下書きに保存する。

### 5.3 記事スタジオ（仕上げる）

- 確定構成に沿って生成された本文を、**セクションごとのカード**としてMarkdownレンダリング表示する（JSONの生表示はしない）。
- セクション単位の操作: 直接編集 / 指示付き再生成（「もっと短く」等）。
- 編集内容は自動保存する。
- クラファン本文はタブ切替で同じ画面から生成・編集できる。

### 5.4 スライド化・挿絵生成

- 記事スタジオの**各セクションカードにチェックボックス**を置き、チェックしたセクションを順次**スライド化**する（セクション本文からタイトル・キーフレーズを抽出し、文字入りスライド画像を生成。文字品質のためhigh画質を強制）。
- 生成されたスライドは該当セクションに紐付けて保存し、エクスポート時に本文内の`<img>`として該当位置に挿入する。
- 記事全体からアイキャッチ画像（イラスト風）も別途生成できる（`eyecatchUrl`用）。
- **画像の「ぼかし→クリア」演出のタイミング設計（重要）**: 生成リクエスト送信と同時にプレースホルダー画像のぼかしを徐々に薄くし始めるが、**実際のAPIレスポンスが届くまでは完全にクリアな状態にしない**（演出が実体験を追い越さないようにする）。レスポンスが届いた瞬間に一気にクリア表示へ切り替える。
- 候補を複数生成して選べるようにする（スライドはコストが高いため、既定1枚+再生成ボタンでもよい）。

### 5.5 書庫（一覧・再開・出口）

- 下書き・記事の一覧を表示し、開けば途中の画面（口述/構成/記事）から再開できる。
- **永続化必須**: インメモリ保存は不可。ローカル開発はSQLite、本番はFirestoreを使い、サーバー再起動でデータが消えないこと。
- 記事画面または書庫から「aiaruku-blogへエクスポート」（8章のJSON形式）。

### 5.6 生成画像の永続化（重要）

- **DALL-E APIが返す画像URLは一時URL（約1時間で失効）**のため、そのまま `generated_images.image_url` に保存してはならない。
- 生成直後にバックエンドで画像をダウンロードして永続化し、永続URLを保存する。
  - 本番: Firebase Storage（aiaruku-blogと同じプロジェクトのStorageを利用可）
  - ローカル開発: `backend/static/images/` に保存し、FastAPIの静的ファイル配信で返す
- ブログの挿絵・アイキャッチとして使うのはこの永続URL。エクスポートJSON（8.2節）の `eyecatchUrl` にも永続URLを入れる。

## 6. バックエンドAPI仕様

| エンドポイント | メソッド | 概要 |
|---|---|---|
| `/drafts` | POST | 下書き新規作成（`raw_text` を受け取る） |
| `/drafts` | GET | 下書き一覧取得（書庫用） |
| `/drafts/{draft_id}` | GET | 下書き詳細取得 |
| `/drafts/{draft_id}/interview` | POST | 沈黙検知トリガーから呼ぶ。未処理の発話断片を受け取り、要点・引用候補の**差分**と、新しい**問いかけ**を返す（5.1節。1回の呼び出しでまとめて生成） |
| `/drafts/{draft_id}/extract` | POST | 要点抽出を実行し `extracted_points` を更新（一括処理用） |
| `/generate/outline` | POST | `draft_id` を受け取り、セクション構成案（タイトル+要旨の配列）を生成（5.2節） |
| `/drafts/{draft_id}/outline` | PUT | 筆者が確定した構成（採用セクション+順序）を保存 |
| `/generate/blog` | POST | `draft_id` と確定構成を受け取り、**構成に従って**本文を生成して `generated_texts` に保存 |
| `/generate/crowdfunding` | POST | 同上、クラファン本文 |
| `/generate/section-slide` | POST | セクション本文を受け取り、スライド画像を生成して該当セクションに紐付け保存（5.4節） |
| `/generate/image` | POST | `prompt`・`image_type` を受け取り画像生成、結果を `generated_images` に保存（アイキャッチ用） |
| `/generate/image/prompt-draft` | POST | 4.4節のたたき台プロンプトを生成して返す（実際の画像生成はしない） |

すべてのエンドポイントは `tenant_id` をクエリパラメータまたはボディで受け取れるようにするが、MVPでは未指定時は `aiaruku` をデフォルト値として扱ってよい。

## 8. ブログ連携（aiaruku-blogへのエクスポート/インポート）

生成したブログ記事は **JSONファイルとしてエクスポート**し、aiaruku-blog側でインポートする。

### 8.1 形式の選定理由（JSON採用、Markdown不採用）

- aiaruku-blogの記事はFirestoreの `posts` コレクション（`Post` 型）で管理されており、構造化されたJSONがそのまま1:1でマップできる
- aiaruku-blogの本文（`Post.content`）はWYSIWYGエディタ出力のHTMLであるため、Markdownで渡してもインポート時にHTML変換が必要になるだけ
- 編集はエクスポート前（VoiceDraft Studio）とインポート後（ブログ管理画面）で行い、転送ファイル自体を人が編集する運用はしない

### 8.2 エクスポートJSON形式

VoiceDraft Studio側でMarkdown→HTML変換を済ませてから出力する。

```json
{
  "source": "voicedraft-studio",
  "exportedAt": "2026-07-02T12:00:00Z",
  "title": "記事タイトル",
  "content": "<p>本文（Markdown→HTML変換済み）</p>",
  "excerpt": "記事の抜粋（先頭150字程度を自動生成、編集可）",
  "tags": ["プログラミング教室"],
  "metaTitle": "",
  "metaDescription": "",
  "eyecatchUrl": "https://.../generated-image.png"
}
```

- `eyecatchUrl` は画像生成画面で選んだ画像の**永続URL**（5.6節）。未生成の場合は空欄とし、ブログ側で設定する
- チェックしたセクションのスライド画像は、`content` 内の該当セクション直下に `<img>` として挿入する（5.4節）
- `slug` / `authorId` などはインポート時にaiaruku-blog側で付与する
- `metaTitle` / `metaDescription` は空欄可（空欄時はブログ側のデフォルト処理に任せる）

**書式の扱い:** 書式はHTMLタグとして `content` に保持される。aiaruku-blogのエディタ（TipTap StarterKit）が対応する書式（見出し・リスト・太字・斜体・引用・コードブロック・リンク・画像）はそのまま維持される。スキーマ外のタグ・属性（表・インラインstyle・色指定など）はエディタで開いた時点で除去されるため、**ブログ生成プロンプトでは表（Markdownテーブル）を使わない**よう指定すること。

### 8.3 インポート（aiaruku-blog側に追加する機能）

- 管理画面にJSONインポート機能を追加し、Firestoreの `posts` に登録する
- **必ず `status: "draft"`（下書き）として取り込む。** 公開はブログ管理画面で人が内容確認・アイキャッチ設定などを行ってから実施する（AIに最終確定させない設計思想と一致）
